## 稼働記録ツール：将来構想（2026-07-11起票）

### 目指す姿

記録 → 集計 → 検証 → 共有 のパイプライン。
記録すること自体が目的ではなく、記録から考える・共有することを楽しめるツールにする。

### 想定利用者（2軸）

**1. 自分用（検証）**
- 狙い目ごとにCZ確率・AT確率を実測する
- 実測値をスペック表（設定別確率）と照合し、打った台の設定帯を推測する
- CZ/AT当選までの投入G数と当選履歴から、その狙い目の期待値がプラスかマイナスかを数値で判断する
- 「なんとなく良さそう」を数値に置き換え、立ち回りを理論的に検証する

**2. SNS勢用（設定判別・考察・共有）**
- 設定示唆の出現割合、当たりやすい場面・CZの偏りなどを検証しやすくする
- 検証結果をワンタップでコピペ共有できる形式にする（X等への投稿）
- 共有された結果が slot-tools.jp の宣伝・流入導線になる

### 機能構想と依存関係

```
ログ条件フィルタ（基盤）
 └→ 集計（CZ/AT確率・示唆出現割合・投入G数・件数）
     ├→ 検証系：設定推測（スペック表照合）・期待値・機械割
     └→ 共有系：コピペ出力・グラフ化

設定推測ツール（小役カウント型）※独立系統・フィルタ非依存
機種別解析クイックリファレンス ※独立系統・機種マスタ基盤を共有
```

#### F-1 ログ条件フィルタ【基盤・最優先】
- 保存済みログを条件で絞り込む（機種／狙い方／期間／当選結果 など）
- 「CZ1スルー後」のような条件は、記録済みのチェーンデータ（CZ当選/AT非突入 等）から機械的に判定する
- 手動タグ付けやメモ検索に頼らない設計（メモは表記ゆれで検索漏れするため、恒常的な条件はデータから自動判定、突発的な情報は自由記述メモ、と役割分担）

#### F-2 集計
- フィルタ結果に対して、CZ確率・AT確率・示唆出現割合・総投入G数・サンプル件数を算出

#### F-3 検証系
- スペック表（設定別CZ/AT確率）との照合による設定帯の目安表示
- 狙い目単位の期待値（プラマイ）表示
- 機械割の算出

#### F-4 共有系
- 集計・検証結果のコピペ出力（SNS投稿用フォーマット、既存の公開/非公開の区分を踏襲）
- グラフ化（示唆割合・確率推移など）

#### F-5 設定推測ツール（小役カウント型）【F-1非依存・単体で成立】
- 設定差のある小役の出現率から、稼働中にリアルタイムで設定を推測する機能（ジャグラーのブドウカウンターと同型）
- 例：モンキーターンの5枚役。機種ごとに設定差のある役が異なるため、全機種対応はせず、必要な機種にだけ順次追加する
- 機種別マスタ：対象小役＋設定別出現率のデータが必要
- 価値：稼働中に外部の推測ツールへ行き来する必要がなくなる（ツール内完結）
- F-1〜F-4のログ集計パイプラインとは独立して動く。ただし将来的には、カウントした小役データをログに残し、事後集計（F-2/F-3）と合流させる余地がある

#### F-6 機種別解析クイックリファレンス（早見表）【F-1非依存】
- 稼働中にワンタップで、その機種の解析情報（モード別規定G数・ゾーン・設定示唆一覧など）を確認できる機能。外部の解析サイトをいちいち開かずにツール内で完結させる
- 実現ルートは2段構え：
  - **ルートB（本命）**：解析値を自前の機種マスタ（JSON等）として再構成し、ツール内で表として描画する。F-3のスペック表マスタ・F-5の設定差役マスタと同じ機種マスタ基盤に乗せる。出典として参照した解析サイトへのリンクを明記する
  - **ルートA（暫定）**：解析サイト運営者（例：ちょんぼりすた）に許可を得て画像・表を表示。**許可必須＋出典リンク必須**。マスタ整備が追いつかない機種の繋ぎとして使う
- ルートBの発展形：規定G数マスタを持てば「現在の液晶G数から次のゾーンまであと◯G」等の能動表示も可能になる
- 設定推測要素（F-3/F-5）と同じ画面で確認できる構成を想定
- 注意：転載コンテンツはAdSense評価・権利面のリスクがあるため、ルートAは許可の書面確認が取れた場合のみ。長期的にはルートBへ寄せる

#### F-7 モーメント共有カード【F-1非依存】
- 「設定6確定が出た」「こんな挙動だった」等の瞬間を、SNS投稿用の共有カード形式で出力する機能
- F-4（検証結果の共有）が数値・考察の共有なのに対し、F-7は感情・体験の共有。どちらもSNS経由の宣伝導線になる
- 出力内容の例：機種／日付／確定演出やハイライト示唆／簡易挙動サマリー＋ツールへのリンク
- 公開/非公開の既存区分（店名・台番号は非公開デフォルト）を必ず踏襲する

#### F-8 ノート装飾・スタンプ【長期・要件未確定】
- 記録する楽しさ・見返す楽しさを増幅する装飾要素（スタンプ、テーマ等）
- 課金化の構想はあるが、現状は静的サイトで決済・アカウント基盤がないため大工事になる。**当面は無料装飾で需要検証のみ**、課金判断はその後
- 生成AI時代にスタンプ絵柄単体の商品価値は疑問。売るなら「体験」（実績バッジ、継続記録の称号等）に寄せる方向を検討

#### F-9 履歴詳細の表示情報フィルタ【F-1と対・構想段階】
- 履歴の展開表示（区切り詳細）で、表示する情報種別をチェックボックスで選べるようにする（例：示唆のみ／モード系のみ／タイムライン非表示／デフォルト・判定対象外を隠す）
- F-1（ログ条件フィルタ）が「どのログを出すか」なのに対し、F-9は「ログ内のどの情報を出すか」。両方揃うと検証時の視認性が完成する
- フェーズ3の階層表示（2026-07-11実装）を使った実戦で要望が発生。具体的な種別分類は実戦使用の蓄積を待って設計する

### 感情設計方針（全機能共通の設計原則）

パチスロにはネガティブ要素が多いため、ツールが負の強化学習を助長しない設計を徹底する。

1. **プロセス強化・結果非強化**：祝う対象は勝ち額・出玉ではなく「期待値のある判断ができた」「記録を継続した」こと。結果ではなく判断の質を評価する（ポーカー的思考）。ツールの核である期待値検証と一致する
2. **負けの処理は構造化ジャーナリング**：自由な愚痴の書き殴りは反芻を強め逆効果になり得るため、構造化された問いで振り返らせる。例：「何が起きた → コントロールできた部分はどこか → 判断は期待値的に正しかったか → 次はどうするか」。負けを「不運（分散）」と「判断ミス」に仕分けることで、感情処理と立ち回り検証を同時に行う。感情に名前を付けること自体に鎮静効果があるという知見（感情ラベリング）、筆記による感情整理の知見（筆記開示）が背景
3. **負の演出をしない**：大負けの強調表示など「痛みの可視化」は行わず、数値は淡々と表示する
4. 将来的にSTEP4のメモ欄を「構造化振り返り」へ進化させる余地がある

### 実装順の考え方

F-1が全ての土台。F-2〜F-4はF-1完成後に検討する。
F-5・F-6はF-1に依存しないため、機種ごとの需要が生じたタイミングで単独着手できる。
F-3・F-5・F-6は「機種マスタ（スペック表・設定差役・規定G数等）」という共通基盤を持つため、マスタのデータ構造は最初に設計しておくと後が楽になる。
F-7はF-4のコピペ出力と実装を共有できる可能性が高い。F-8は長期枠。
着手タイミングは、保存済みログ画面がある程度育ってから（F-1系）、または対象機種を打つ予定が立ったとき（F-5/F-6）。

### 前提・注意

- 既存の3層構造（稼働セッション／狙い単位／区切り記録）とlocalStorage（nerai_record_v1）を集計の情報源とする。構想実装時もデータモデルの破壊的変更は避ける
- スペック表データ（設定別確率）は機種ごとのマスタが必要になる。東京喰種から始める
