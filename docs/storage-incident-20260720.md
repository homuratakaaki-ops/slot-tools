# storage incident 2026-07-20

## 1. 事故の経緯

2026-07-20 頃、`nerai-record.html` の実戦記録で localStorage 使用量が大きく膨らみ、保存容量の上限に近づいた。

容量逼迫により保存処理が失敗し、画面上では操作継続できているように見えても、内部では保存が完了していない状態が発生し得る状況だった。結果として、ログ・下書き・退避データの一部消失につながる危険があった。

消失後にページを開き直さず、`recovery.html` 経由で退避データを確認したことで、復旧可能な JSON を確保できた。

## 2. 根本原因

直接の肥大化原因は `suggestLog` の累積コピーだった。

区切り保存時、`createSegmentSnapshot()` が現在の `currentSuggestLog` 全体を `segment.suggestLog` に保存していた。区切り保存後、持ち越し対象の示唆は次区切りへ残るため、次の区切りでも同じ示唆が再び保存される。

このため、CZ/AT 終了、当選前ヤメ、区切り保存などを繰り返すと、持ち越し示唆が複数区切りへ重複保存された。南国ログでは全 50 区切りに近い範囲で同一内容が複製され、1 ログが約 144 万文字まで肥大化していた。

## 3. 復旧手順の記録

復旧は以下の流れで行った。

1. 問題発覚後、通常ページを開き直さず、追加保存を避けた。
2. `recovery.html` を使い、localStorage 内の主キー・退避キーを確認した。
3. 回収 JSON を保存した。
4. `premigrate` 由来の JSON から重複 `suggestLog` を dedup した復旧用 JSON を作成した。
5. 復旧後 JSON を再保存し、実データが戻っていることを確認した。

復旧時点で、回収 JSON と復旧後 JSON の二重バックアップを確保した。

## 4. 実装した対策

`ecf17d1` 以降、以下の対策を実装した。

- 区切り保存時は、現在区切りに属する示唆のみを `segment.suggestLog` に保存する。
- 過去バックアップを読み込んだ場合も、同一ログ内の区切り間 `suggestLog` 重複を正規化で除去する。
- `storageCounts()` に `shopNotes` と `draftLog` を含め、部分消失を検知しやすくした。
- `logs 1 -> 0`、`draftLog` 消失、`shopNotes` 大幅減少、`sessionId` 消失を危険保存として停止する。
- `QuotaExceededError` を捕捉し、自動保存を保護ロックへ切り替える。
- localStorage 使用量を概算し、60% でバックアップとアーカイブ推奨、80% で保存失敗危険の警告を表示する。
- 復元前に `prerestore` 退避と個別 JSON 保存を行う。
- 退避データ一覧から不要な退避キーを個別削除できるようにした。削除前には対象 JSON を保存し、キー名と文字数を確認する。

## 5. 運用ルール

通常運用では、実戦前または大きな編集前に JSON バックアップを保存する。

使用量が 60% を超えたら、不要な退避データを整理し、古いログを必要に応じて外部 JSON として保管する。

使用量が 80% を超えたら、そのまま実戦を続けない。先に JSON バックアップを保存し、`premigrate`、`prerestore`、古い `backup_1` など不要になった退避キーを削除して容量を下げる。

保存失敗、保護バナー、復元警告が出た場合は、通常ページで追加操作を続けず、まず `recovery.html` またはバックアップ/復元エリアから JSON を保存する。

同種の事故が再発した場合は、ページを開き直す前に退避 JSON を確保し、主キー・`last_good`・`premigrate`・`prerestore`・日次バックアップの順に内容を比較する。
