## 0. 目的と位置づけ

nerai-record.html の実戦中専用入力ビュー「実戦モード」の仕様。
既存の稼働入力画面は「管理・編集画面」として存続し、記録の入口を実戦モードが、詳細補完・修正の出口を既存画面が担う。

設計方針: 実績ある現行実戦メモ画面のレイアウトを構造として固定し、取り消しバー・振動フィードバックを追加、管理系UIを排除したもの。頻度と時間圧に応じて操作コストを配分する（高頻度・高時間圧＝1タップ、低頻度・低時間圧＝多段入力を許容）。

### ゴール指標（実機検証済み）

| 指標 | 目標 | 結果 |
|---|---|---|
| カウンター系イベント記録 | 1タップ | 達成 |
| ピッカー系イベント記録 | 2〜3タップで記録完了 | 達成 |
| 誤入力からの復帰 | 1タップ | 達成 |
| 実戦モード内のスクロール | 原則0回 | 達成 |
| 保存確認 | トースト＋振動、確定ボタンなし | 達成 |
| G数表示と主要操作の同一画面 | 常時 | 達成 |

## 1. スコープ

### 骨格（実装済み）

1. **画面分離**: 実戦モード専用ビュー（全画面オーバーレイ）。管理系UI・集計確定系は非表示
2. **固定配置**: §2 のゾーン構成を固定。状態でボタン位置を動かさない（無効時はグレーアウトのみ）
3. **タップ即保存**: 確定操作なしで localStorage へ即書き込み
4. **常時取り消し**: 取り消しバーを画面下部に常駐。復元（Undo の Undo）対応
5. **統一ピッカー**: さざなみ・ボーナスの2つ
6. **同一画面化**: G数表示と主要操作が常に同一ビューポート内

### 非スコープ

- クイックボタンの並べ替え・表示切替のユーザー編集UI（§9）
- 新しいログ種別・データ層の追加（既存4層構造・timeline形式を使用）
- 逐次チップ・割り込み復帰・進行シーケンス・自動推測・Wake Lock 等（§9）

## 2. 画面構成

### 2.1 モード切替

- 稼働入力タブ内、アクティブな稼働セッション存在時に「実戦モードで記録」ボタンを表示
- 実戦モードは全画面オーバーレイ。右上「✕」で既存画面へ戻る。**閉じてもセッション状態・入力内容は変化しない**（表示切替のみ）

### 2.2 ゾーン構成（v1.3 確定版・上から順）

```text
┌──────────────────────────────┐
│ 機種名 / 狙い方名           ✕ │ ← ヘッダ
├──────────────────────────────┤
│ データG数 [184]  液晶G数 [176] │ ← G数表示（タップで既存テンキー）
│ 差分 [0] [+9]   スイカ [0] [+1]│ ← カウンター操作（現行踏襲）
│ 差枚 [+120]                    │ ← 有利区間差枚（小型・タップ可）
├──────────────────────────────┤
│ [プリリプ] [さざなみ] [ボーナス]│ ← イベント段
├──────────────────────────────┤
│ [その他 ▼]                    │
├──────────────────────────────┤
│ [リプレイ] [チェリー] [スイカ] │ ← カウンター段（最下段ボタン行）
├──────────────────────────────┤
│ [＋1]    [＋5]    [＋10]      │ ← G加算段（最下段ボタン行）
├──────────────────────────────┤
│ 直近: 184G さざなみ赤 14:32    │ ← 直近1〜3件（読み取り専用）
├──────────────────────────────┤
│ ↶ 直前「184G スイカ」を取消    │ ← 取り消しバー（常駐・最下部）
└──────────────────────────────┘
```

- **行順の根拠（v1.3改訂）**: 操作するものは下半分、頻度が高いほど下、表示するだけのものは上に置く。最頻操作のG加算を最下段へ、リプレイ/チェリー/スイカをその直上へ、その他を中間頻度としてイベント段とカウンター段の間へ配置する
- **記録トースト（v1.2改訂）**: 記録成功・取り消し・無効操作のトーストは**画面上部中央**（G数表示付近の視線位置）に表示する。実戦中の視線は画面上半分にあるため。取り消しバーは下部常駐のまま
- 現行実戦メモ画面との差分 — 追加: 取り消しバー・直近表示・振動フィードバック / 削除: 「確定にする」「示唆記録」大ボタン等の管理・確定系、さざなみ3色個別ボタン、リプレイフラッシュ個別ボタン / 統合: BIG/REG→ボーナスピッカー、さざなみ3色→さざなみピッカー
- 示唆記録は「その他 ▼」から既存モーダルを呼ぶ形で存続

### 2.3 寸法基準

| 項目 | 値 |
|---|---|
| ボタン高さ | 52〜64px |
| ボタン間隔 | 6〜10px |
| ボタン文字 | 16px以上 |
| G数数値 | 20〜28px |
| タップ領域 | 最低48×48px |

縦に収まらない端末のみ直近表示を1件に減らして吸収。ボタンサイズは削らない。

## 3. ボタン仕様

### 3.1 ボタン型の種類

| type | 挙動 |
|---|---|
| counter | タップで既存カウンター処理を実行し即保存。**現行実戦メモ画面の同名ボタンと同一の既存処理を呼ぶ**（スイカ＝タグ記録＋カウンター加算、リプレイ／チェリー＝タグ記録のみ） |
| tag | タップで該当タグを timeline へ即時記録 |
| picker | タップで選択シートを開き、選択完了で記録（さざなみ／ボーナス） |
| modal | 既存モーダルを開くだけ（示唆記録・他台メモ等） |
| hit | タップで既存の当選記録ウィザードを起動（§3.5） |

### 3.2 「その他 ▼」（v1.2確定版）

タップでボトムシートを開き、以下を表示する。

1. その機種の全タグのうち、**quickPanel（counters／events）に含まれない項目のみ**（リプレイフラッシュ・低頻度タグ等）。quickPanel未定義機種では全タグ表示
2. **他台メモ**: タップで既存の他台メモモーダル（openShopNoteModal）を開く（modal型。既存モーダルの挙動は不変）
3. **メモ**: タップでメモシート（§3.7）を開く
4. 示唆記録モーダルへの導線

一覧からのタグタップも即時記録。

### 3.3 さざなみピッカー

- 「さざなみ」タップでボトムシート
- ステップ1（契機・6択）: プリリプ／スイカ3回目／スイカ5回目／スイカ7回目／スイカ9回目／それ以外
- ステップ2（色・3択）: 紫／赤／虹
- 色の選択と同時に記録・保存しシートを閉じる（確定ボタンなし）
- 当選有無は扱わない（当選はボーナスピッカー／hitTriggers の責務）
- 保存形式: 既存の さざなみ紫/赤/虹 の3タグとして timeline へ保存し、契機を entryCause / entryCauseLabel に格納（§4.5）。新タグは作らない
- 画面外タップまたは「閉じる」で何も記録せずキャンセル

### 3.4 ボーナスピッカー

- 「ボーナス」タップでボトムシート、1ステップ・3択: 赤BIG／青BIG／REG
- 選択と同時に対応する当選記録ウィザードを起動（§3.5）
- 対応表（L南国）:

| 選択肢 | 内部 trigger | variant | 備考 |
|---|---|---|---|
| 赤BIG | direct_at | なし | variant無し＝赤BIG。過去ログとの連続性維持 |
| 青BIG | direct_at | blue7 | ウィザードに variant 選択済みで入る |
| REG | episode_bonus | なし | |

- STEP3側の青7チップUIは現状維持（実戦モード外からの記録・修正用）

### 3.5 hit型の挙動（v1.2追記あり）

- タップ（ピッカー選択）で `startHitEventWizard('<trigger>')` 相当の既存の当選記録フローを起動。以降は既存フロー（AT当選ルート→獲得枚数→終了時示唆等）に従う。新しい入力経路・省略経路は作らない
- 設計意図: カウンター系は遊技中の高頻度操作のため1タップ、当選系はボーナス消化中で時間圧が低いため既存の流れで示唆まで完結させる（獲得枚数・終了時示唆の後回しは記録漏れの原因になるため）
- ウィザード起動時は実戦モードオーバーレイをいったん閉じてSTEP3へ渡す。**ウィザード通常完了後およびルート解除による中断後、実戦モードへ自動復帰する**
- **STEP3側の誘導（v1.2追記）**: ボーナスピッカー経由の遷移直後のみ、①「次の入力：◯◯」ボタンが見える位置まで自動スクロール、②同ボタンをタップ領域48px以上に大型化し主要ボタンとして強調。ウィザードのフロー構造・入力項目・例外操作（青7チップ、ルート解除等）は不変。実戦モード外からSTEP3を直接使う場合の表示・挙動に副作用を与えない

### 3.6 L南国育ちSPECIAL quickPanel（確定版）

```js
quickPanel: {
  counters: ["replay", "cherry", "suika"],
  gainButtons: [1, 5, 10],
  events: [
    { type: "tag",    ref: "t_nangoku_puririp" },
    { type: "picker", ref: "sazanami" },
    { type: "picker", ref: "bonus" }
  ]
}
```

- counters は「カウンター段に出す1タップ記録ボタン」の定義。各ボタンの挙動は §3.1 counter型のとおり現行同一処理
- quickPanel は配列形式（v1.0互換）とオブジェクト形式の両方を正規化して受け付ける
- quickPanel 未定義の機種では、G加算段・「その他 ▼」のみで実戦モードを成立させる（イベント段・カウンター段は空欄表示）

### 3.7 メモシート（v1.2新設）

- 「その他 ▼」→「メモ」で開くボトムシート。テキスト入力欄＋追加／戻るボタン
- シート表示と同時に入力欄へキーボードフォーカスを当てる
- 保存は既存の timeline 自由記述と同形式（tagIds: [] の通常 timeline entry）。保存処理は addTimelineEntryText() に共通化されており、既存STEP2のメモ追加も同経路
- **追加後はシートを閉じて実戦モードへ戻る**（連続入力モードは設けない）
- **キーボード追従**: キーボード表示中も入力欄と追加ボタンが常に可視領域に収まること。visualViewport の resize/scroll、visibilitychange、window focus、pageshow、フォーカス時・入力時に位置再計算を行う（Android Chrome のタスク切り替え復帰後に再計算が走らず、シートがキーボード裏に隠れる事象への対策）

### 3.8 有利区間差枚トラッカー（v1.3新設）

- G数表示ブロック内に「差枚 +120」形式で小型表示する。投資・クレジット・初期差枚がすべて未入力の場合は「差枚 —」を表示
- 表示部タップでボトムシートを開く
- シートには現在値と内訳（初期 + クレジット - 投資）を表示し、以下の操作を置く
- **投資追加（千円）**: 入力値 × intervalEstimate.loanRate を四捨五入し、investedTotal へ加算
- **投資追加（枚）**: 入力値を investedTotal へ加算
- **クレジット更新**: credit を入力値で上書き
- **初期差枚設定**: initialDiff を入力値で上書き（±入力可）
- **差枚を直接補正**: 入力値を現在の有利区間差枚として扱い、initialDiff を `入力値 - (credit - investedTotal)` で逆算して上書き
- 各操作は完了時に即保存し、シート上の現在値表示を更新する
- certainty は実戦モード側から読み書きしない

## 4. 記録フロー（タップ即保存）

### 4.1 基本フロー

1. ボタンタップ（picker は選択完了時点）
2. 既存のログ追加処理を呼び、現在のセッション・区切りに紐づけて即時に localStorage へ保存
3. フィードバック（§5）
4. 直近表示と取り消しバーを更新

確定ボタン・確認ダイアログは挟まない（hit型のウィザード内は既存仕様に従う）。

### 4.2 記録内容

- counter/tag/picker のタップ時点で保存するのは 時刻・現在G数・タグ種別（＋picker の選択内容）
- 数値やメモの補完は既存ログ画面での編集に委ねる

### 4.3 G数加算（＋1／＋5／＋10）

- タップで現在G数に加算し即保存。既存のG数管理に従う。取り消しバーの対象に含める

### 4.4 既存ロジックとの関係

- ログの生成・保存・区切りへの紐づけは既存関数を呼ぶ。実戦モード側でログ構造を再定義しない
- スイカオフセット2層管理・セグメントライフサイクルに関わる処理は既存経路をそのまま通す

### 4.5 さざなみ契機の付帯情報（entryCause の意味拡張）

- さざなみタグの timeline entry に契機を entryCause / entryCauseLabel で格納
- 意味拡張: entryCause は従来「SDC/超飛翔突入の契機」用途。本仕様により「timelineタグの契機付帯」へ拡張
- 値の列挙（固定・接頭辞 `sazanami_` 必須）:

| entryCause | entryCauseLabel |
|---|---|
| sazanami_pururipu | プリリプ |
| sazanami_suika3 | スイカ3回目 |
| sazanami_suika5 | スイカ5回目 |
| sazanami_suika7 | スイカ7回目 |
| sazanami_suika9 | スイカ9回目 |
| sazanami_other | それ以外 |

- 既存値（tokimeki32／bonus1g／freeze／unknown）と名前空間分離済み・衝突なしを確認済み
- さざなみには entryCause / entryCauseLabel のみ付与し、intervalEvent・entryType には触れない（SDC/超飛翔系との混線防止）
- attachments は value 数値必須のため本用途には使用しない

### 4.6 有利区間差枚トラッカーの保存形式（v1.3新設）

狙い（セッション）ごとに、既存の `intervalEstimate` を以下の形へ拡張して保持する。新しい保存場所は作らない。

```js
intervalEstimate: {
  initialDiff,    // 既存。null許容。明示操作時のみ数値化
  certainty,      // 既存。実戦モード側からは読み書きしない
  loanRate,       // 既存。投資追加（千円）の換算に使用
  investedTotal,  // v1.3追加。累計投資枚数。既定0
  credit          // v1.3追加。現在クレジット枚数。既定0
}
```

計算式: `有利区間差枚 = (initialDiff ?? 0) + (credit - investedTotal)`

- 既存STEP2の「未設定」表示を壊さないため、保存上は initialDiff の null を維持する
- 実戦モードの表示・計算時のみ `initialDiff ?? 0` として扱う
- 投資・クレジットがともに0で、initialDiff も null の場合は未計測表示とする
- 既存STEP2の有利区間差枚メモ、timeline の `intervalEvent:'invest'/'credit'`、判定ロジックには触れない

## 5. フィードバック仕様（v1.2改訂）

| イベント | 振動 | 表示 |
|---|---|---|
| 記録成功 | 短1回（10〜30ms） | ボタン押下アニメ＋トースト「◯◯を記録」0.8秒 |
| 取り消し | やや長め1回 | トースト「◯◯を取り消し［元に戻す］」 |
| 無効操作 | 短2回 | トースト（理由を1行） |

- **トーストは画面上部中央に表示**（既存STEP2の記録トーストと同位置・同等の見た目。v1.1の「取り消しバー直上」から変更）
- `navigator.vibrate` を使用。非対応端末では表示のみで成立。音は使用しない

## 6. 取り消し仕様

- 取り消しバーは常駐（画面下部）。直前イベントの内容を表示
- タップで直前イベントを削除。確認ダイアログは出さない
- 削除後、トースト内の［元に戻す］で復元可能（最低5秒）
- スタック管理で連打により複数件さかのぼれる（上限は直近の区切り内まで）
- 対象: counter（G加算含む）、tag、picker で記録したイベント、有利区間差枚トラッカー操作（投資追加・クレジット更新・初期差枚設定・直接補正）
- 対象外: hit型ウィザードで確定した当選記録、区切りをまたぐ削除、確定済み収支（無効操作フィードバックを返し、既存の編集経路で修正させる）
- G数加算・カウンターの取り消しは加算前の値へ戻す（スイカ履歴・オフセットとの整合を含む）

### 6.5 ログ画面の区切り折りたたみ（v1.2追記）

- ログ画面の各区切り（初当たり単位のブロック）はヘッダタップで開閉可能
- ヘッダには既存のサマリ行（「前回：BIG当選／最終292/292G」等）と区切りバッジ（1-1等）を維持し、閉じた状態で内容の要約が分かること
- 初期状態: **当日の稼働の最新区切りのみ開、それ以外はすべて閉**（当日以外の日付の稼働は全区切り閉）
- 「編集」ボタンはヘッダ右端に配置し、開閉状態によらず同じ位置で押せること
- 開閉状態の保持はセッション内で十分（リロード後は初期状態に戻ってよい）
- 整理ビュー・PNGカード等、他の出口には影響しない

## 7. 既存コードへの影響範囲（ミコトへの制約）

**変更してよい範囲**

- 実戦モード用のビュー・JSモジュール（接頭辞 `bm` 等で名前空間分離）
- 稼働入力タブの「実戦モードで記録」ボタン
- 機種プリセットの quickPanel フィールド（後方互換: 未定義でも既存動作に影響しないこと）
- さざなみタグ保存時の entryCause / entryCauseLabel 付与（§4.5 の列挙値のみ）
- 有利区間差枚トラッカー用の intervalEstimate 拡張フィールド（§4.6 の investedTotal / credit のみ）

**変更してはいけない範囲**

- 既存のログデータ構造・localStorage スキーマ（§4.5・§4.6 の範囲を除く。追加フィールドは事前相談）
- 既存モーダル・当選記録ウィザードの内部挙動（呼び出し、variant 初期選択の受け渡し、§3.5 の誘導強化の範囲を除く）
- スイカオフセット・セグメントライフサイクル・エンディング集計・PNG共有カードの各ロジック
- 既存の稼働入力画面のDOM構造（ボタン追加箇所を除く）

**実装上の注意（v1.2追記）**

- 実戦モードオーバーレイより手前に出すべき既存モーダル（テンキー等）の z-index はオーバーレイ（9000）より高くすること（テンキーは 9200 で対応済み。新規モーダル呼び出しを追加する際は同様の重なり順を確認する）
- キーボードと共存するシートは §3.7 のとおり visualViewport 追従と復帰時再計算を行うこと

機種プリセットに触れる場合は docs/machine-preset-guide.md の参照と3系統検証を必ず実施。

## 9. 非スコープ（IDEAS控え）

- **配置カスタムUI**（クイックボタンの並べ替え・表示切替。実戦での親指可動域は個人差が大きく、固定配置の最適解は存在しないことが実測で確認されたため。**優先度上げ**。最楽ゾーン4枠＋機種別最頻タグ1枠を各自が構成するモデルを検討）
- **有利区間差枚の履歴グラフ**（区間ごとの推移記録・グラフ化。v1.3では現在値の表示と記録のみ）
- **ボーナス一括入力シート（B案）**（ボーナス選択→獲得枚数→終了時示唆を実戦モードのボトムシート連鎖で完結させる。STEP3ウィザードの再実装に近い中規模改修のため、実戦データを蓄積し必要項目が定まってから仕様化する。当座はA案＝§3.5の誘導強化で対応済み）
- 逐次チップ（記録直後の詳細チップ表示）
- 割り込み復帰（離脱→復帰時のG数更新プロンプト）
- 確認省略/即時記録のモード切替設定
- 進行シーケンス（区切りチェーンの事前定義進行）
- 長押し詳細入力・最近使用履歴・Wake Lock・自動推測
- リプレイ等カウンター実測値を使った分母分析の強化
