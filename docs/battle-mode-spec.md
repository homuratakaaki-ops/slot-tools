## v1.2 → v1.5 変更概要

実戦モードの骨格運用、自宅検証、ホール実戦、AQUOS Sense9実機確認を経て確定した変更を統合する。

1. ボタン行順を最終化: `＋1/＋5/＋10 → リプレイ/チェリー/スイカ → リプレイフラッシュ/さざなみ前兆/ボーナス → その他`
2. L南国 quickPanel のイベント枠を `プリリプ` から `リプレイフラッシュ` へ変更し、さざなみは2行ラベル「さざなみ/前兆」にした
3. その他シートに近接「閉じる」を追加し、プリリプ/パトランプ発光/スルーを実戦モードのその他シートから除外した
4. 有利区間差枚トラッカーを追加し、単位表示、換算プレビュー、現在値表示、投資合計修正、履歴、任意のログ記録に対応した
5. 差枚履歴を `intervalEstimate.history` へ永続化した。保存上限50件、表示10件、個別取消なし
6. ヤメUIを `terminalType: "follow_miss"` に一本化し、STEP4遷移時と新規稼働開始時に未終端区切りを閉じるガードを追加した
7. 設計原則として「数値入力には単位・換算結果・現在値を添える」を明記した

## 0. 目的と位置づけ

nerai-record.html の実戦中専用入力ビュー「実戦モード」の仕様。
既存の稼働入力画面は「管理・編集画面」として存続し、記録の入口を実戦モードが、詳細補完・修正の出口を既存画面が担う。

設計方針:

- 頻度と時間圧に応じて操作コストを配分する。高頻度・高時間圧は1タップ、低頻度・低時間圧は多段入力を許容する
- 操作するものは親指の可動域に置き、表示だけのものは上部へ寄せる
- 数値入力には、単位・換算結果・現在値を添える。入力者に暗算や現在値記憶を要求しない
- 記録の自動転送は増やしすぎない。分析上残したい瞬間だけ、明示ボタンでtimelineへ残す

### ゴール指標（v1.5時点で実機確認済み）

| 指標 | 目標 | 結果 |
|---|---|---|
| カウンター系イベント記録 | 1タップ | 達成 |
| ピッカー系イベント記録 | 2〜3タップで記録完了 | 達成 |
| 誤入力からの復帰 | 1タップ | 達成 |
| 実戦モード内のスクロール | 原則0回 | 達成 |
| 保存確認 | トースト＋振動、確定ボタンなし | 達成 |
| G数表示と主要操作の同一画面 | 常時 | 達成 |
| 差枚トラッカーの実戦入力 | 単位・換算・現在値を見ながら入力 | 達成 |

## 1. スコープ

### 実装済み

1. **画面分離**: 実戦モード専用ビュー（全画面オーバーレイ）。管理系UI・集計確定系は非表示
2. **固定配置**: §2 のゾーン構成を固定。状態でボタン位置を動かさない（無効時はグレーアウトのみ）
3. **タップ即保存**: 確定操作なしで localStorage へ即書き込み
4. **常時取り消し**: 取り消しバーを画面下部に常駐。復元（Undo の Undo）対応
5. **統一ピッカー**: さざなみ・ボーナスの2つ
6. **有利区間差枚トラッカー**: 実戦モード内で投資・クレジット・初期差枚を管理
7. **ヤメUI一本化**: 当選前ヤメを区切り終端 `follow_miss` として扱う

### 非スコープ

- クイックボタンの並べ替え・表示切替のユーザー編集UI（§9）
- 新しいログ種別・データ層の追加（既存4層構造・timeline形式を使用）
- 差枚履歴グラフ、差枚を使った自動判断
- 逐次チップ・割り込み復帰・進行シーケンス・自動推測・Wake Lock 等（§9）

## 2. 画面構成

### 2.1 モード切替

- 稼働入力タブ内、アクティブな稼働セッション存在時に「実戦モードで記録」ボタンを表示
- 実戦モードは全画面オーバーレイ
- 右上「×」で既存画面へ戻る。閉じてもセッション状態・入力内容は変化しない（表示切替のみ）

### 2.2 ゾーン構成（v1.5確定版・上から順）

```text
┌──────────────────────────────┐
│ 機種名 / 狙い方名           × │ ← ヘッダ
├──────────────────────────────┤
│ データG数 [184]  液晶G数 [176] │ ← G数表示（タップで既存テンキー）
│ 差分 [0] [+9]   スイカ [0] [+1]│ ← カウンター操作（現行踏襲）
│ 差枚 [+120]                    │ ← 有利区間差枚（小型・タップ可）
├──────────────────────────────┤
│ [＋1]    [＋5]    [＋10]      │ ← G加算段
├──────────────────────────────┤
│ [リプレイ] [チェリー] [スイカ] │ ← カウンター段
├──────────────────────────────┤
│ [リプレイフラッシュ] [さざなみ]│ ← イベント段
│                  [前兆] [ボーナス]│
├──────────────────────────────┤
│ [その他 ▼]                    │
├──────────────────────────────┤
│ 直近: 184G さざなみ赤 14:32    │ ← 直近1〜3件（読み取り専用）
├──────────────────────────────┤
│ ↶ 直前「184G スイカ」を取消    │ ← 取り消しバー（常駐・最下部）
└──────────────────────────────┘
```

- 行順の根拠: 実戦で親指の自然位置を検証した結果、最頻のG加算は最下段ではなく1段上が最も安定した
- G加算段、カウンター段、イベント段、その他の順に頻度と時間圧を下げている
- 記録トーストは画面上部中央に表示する。取り消しバーは下部常駐のまま
- 示唆記録は「その他 ▼」から既存モーダルを呼ぶ形で存続

### 2.3 寸法基準

| 項目 | 値 |
|---|---|
| ボタン高さ | 52〜64px |
| イベント段ボタン高さ | 2行ラベルが収まる高さで統一 |
| ボタン間隔 | 6〜10px |
| ボタン文字 | 16px以上 |
| G数数値 | 20〜28px |
| タップ領域 | 最低48×48px |

縦に収まらない端末のみ直近表示を1件に減らして吸収。ボタンサイズは削らない。

## 3. ボタン仕様

### 3.1 ボタン型の種類

| type | 挙動 |
|---|---|
| counter | タップで既存カウンター処理を実行し即保存。現行実戦メモ画面の同名ボタンと同一の既存処理を呼ぶ（スイカ＝タグ記録＋カウンター加算、リプレイ／チェリー＝タグ記録のみ） |
| tag | タップで該当タグを timeline へ即時記録 |
| picker | タップで選択シートを開き、選択完了で記録（さざなみ／ボーナス） |
| modal | 既存モーダルを開くだけ（示唆記録・他台メモ等） |
| hit | タップで既存の当選記録ウィザードを起動（§3.5） |

### 3.2 「その他 ▼」

タップでボトムシートを開き、以下を表示する。

1. 近接「閉じる」ボタン（タグ一覧グリッドの先頭左）。右上の既存「閉じる」も残す
2. その機種の全タグのうち、quickPanel（counters／events）に含まれない項目
3. 他台メモ: 既存の他台メモモーダル（openShopNoteModal）を開く
4. メモ: メモシート（§3.7）を開く
5. 示唆記録モーダルへの導線

L南国育ちSPECIALでは、実戦モードのその他シートから以下を固定除外する。タグマスタや過去ログ表示からは削除しない。

| tagId | ラベル | 除外理由 |
|---|---|---|
| t_nangoku_puririp | プリリプ | さざなみピッカーの契機選択肢に含まれるため |
| t_nangoku_pato_light | パトランプ発光 | 実戦モードの低頻度ノイズになったため |
| t_nangoku_through | スルー | 南国seed初期タグだが実戦運用で使用しないため |

### 3.3 さざなみピッカー

- ボタン表示は2行ラベル「さざなみ / 前兆」
- ステップ1（契機・6択）: プリリプ／スイカ3回目／スイカ5回目／スイカ7回目／スイカ9回目／それ以外
- ステップ2（色・3択）: 紫／赤／虹
- 色の選択と同時に記録・保存しシートを閉じる（確定ボタンなし）
- 保存形式: 既存の さざなみ紫/赤/虹 の3タグとして timeline へ保存し、契機を entryCause / entryCauseLabel に格納（§4.5）。新タグは作らない

### 3.4 ボーナスピッカー

- 「ボーナス」タップでボトムシート、1ステップ・3択: 赤BIG／青BIG／REG
- 選択と同時に対応する当選記録ウィザードを起動（§3.5）

| 選択肢 | 内部 trigger | variant | 備考 |
|---|---|---|---|
| 赤BIG | direct_at | なし | variant無し＝赤BIG。過去ログとの連続性維持 |
| 青BIG | direct_at | blue7 | ウィザードに variant 選択済みで入る |
| REG | episode_bonus | なし | |

### 3.5 hit型の挙動

- タップ（ピッカー選択）で `startHitEventWizard('<trigger>')` 相当の既存の当選記録フローを起動
- 起動後は既存フロー（AT当選ルート→獲得枚数→終了時示唆等）に従う。新しい入力経路・省略経路は作らない
- ウィザード起動時は実戦モードオーバーレイをいったん閉じてSTEP3へ渡す
- ウィザード通常完了後およびルート解除による中断後、実戦モードへ自動復帰する
- ボーナスピッカー経由の遷移直後のみ「次の入力」ボタンが見える位置まで自動スクロールし、同ボタンをタップ領域48px以上に大型化する

### 3.6 L南国育ちSPECIAL quickPanel（v1.5確定版）

```js
quickPanel: {
  counters: ["replay", "cherry", "suika"],
  gainButtons: [1, 5, 10],
  events: [
    { type: "tag",    ref: "t_nangoku_replay_flash" },
    { type: "picker", ref: "sazanami" },
    { type: "picker", ref: "bonus" }
  ]
}
```

- `t_nangoku_replay_flash` は既存タグ「リプレイフラッシュ」。新タグは作らない
- quickPanel は配列形式（v1.0互換）とオブジェクト形式の両方を正規化して受け付ける
- quickPanel 未定義の機種では、G加算段・「その他 ▼」のみで実戦モードを成立させる

### 3.7 メモシート

- 「その他 ▼」→「メモ」で開くボトムシート。テキスト入力欄＋追加／戻るボタン
- シート表示と同時に入力欄へキーボードフォーカスを当てる
- 保存は既存の timeline 自由記述と同形式（tagIds: [] の通常 timeline entry）
- 保存処理は addTimelineEntryText() に共通化されており、既存STEP2のメモ追加も同経路
- 追加後はシートを閉じて実戦モードへ戻る
- キーボード表示中も入力欄と追加ボタンが可視領域に収まるよう visualViewport 追従と復帰時再計算を行う

### 3.8 有利区間差枚トラッカー

- G数表示ブロック内に「差枚 +120」形式で小型表示する
- 投資・クレジット・初期差枚がすべて未入力の場合は「差枚 —」を表示
- 表示部タップでボトムシートを開く
- シートには現在値、内訳、操作ボタン、履歴、ログ記録ボタンを表示する

操作:

| 操作 | 挙動 |
|---|---|
| 投資追加（千円） | 入力値 × intervalEstimate.loanRate を四捨五入し、investedTotal へ加算 |
| 投資追加（枚） | 入力値を investedTotal へ加算 |
| 投資合計を修正 | investedTotal を入力値で上書き |
| クレジット更新 | credit を入力値で上書き |
| 初期差枚設定 | initialDiff を入力値で上書き（±入力可） |
| 差枚を直接補正 | 入力値を現在の有利区間差枚として扱い、initialDiff を `入力値 - (credit - investedTotal)` で逆算して上書き |
| ログに記録 | 現在の差枚スナップショットを timeline メモとして保存 |

数値入力:

- 投資追加（千円）は「千円単位（1 = 1,000円）」を明記し、`5 → 5,000円 → 233枚` のように換算プレビューを強調表示する
- 投資合計を修正／クレジット更新／初期差枚設定／差枚を直接補正では、「現在: 1,010枚」のように対象の現在値を表示する
- certainty は実戦モード側から読み書きしない

ログに記録:

- 自動転送はしない
- ボタン押下時のみ、現在G数付きの timeline メモとして保存する
- 例: `差枚 +490（初期 0 ＋ クレ 1500 − 投資 1010）`

## 4. 記録フロー（タップ即保存）

### 4.1 基本フロー

1. ボタンタップ（picker は選択完了時点）
2. 既存のログ追加処理を呼び、現在のセッション・区切りに紐づけて即時に localStorage へ保存
3. フィードバック（§5）
4. 直近表示と取り消しバーを更新

確定ボタン・確認ダイアログは挟まない（hit型のウィザード内は既存仕様に従う）。

### 4.2 記録内容

- counter/tag/picker のタップ時点で保存するのは、時刻・現在G数・タグ種別（＋picker の選択内容）
- 差枚トラッカーの操作は intervalEstimate に保存する
- 差枚トラッカーの「ログに記録」は、現在値スナップショットを addTimelineEntryText() 相当の経路で timeline メモとして保存する

### 4.3 G数加算（＋1／＋5／＋10）

- タップで現在G数に加算し即保存。既存のG数管理に従う
- 取り消しバーの対象に含める

### 4.4 既存ロジックとの関係

- ログの生成・保存・区切りへの紐づけは既存関数を呼ぶ。実戦モード側でログ構造を再定義しない
- スイカオフセット2層管理・セグメントライフサイクルに関わる処理は既存経路をそのまま通す

### 4.5 さざなみ契機の付帯情報（entryCause の意味拡張）

- さざなみタグの timeline entry に契機を entryCause / entryCauseLabel で格納
- entryCause は従来「SDC/超飛翔突入の契機」用途。本仕様により「timelineタグの契機付帯」へ拡張
- 値は `sazanami_` 接頭辞で名前空間を分離する

| entryCause | entryCauseLabel |
|---|---|
| sazanami_pururipu | プリリプ |
| sazanami_suika3 | スイカ3回目 |
| sazanami_suika5 | スイカ5回目 |
| sazanami_suika7 | スイカ7回目 |
| sazanami_suika9 | スイカ9回目 |
| sazanami_other | それ以外 |

- さざなみには entryCause / entryCauseLabel のみ付与し、intervalEvent・entryType には触れない
- attachments は value 数値必須のため本用途には使用しない

### 4.6 有利区間差枚トラッカーの保存形式

既存の `intervalEstimate` を以下の形へ拡張して保持する。新しい保存場所は作らない。

```js
intervalEstimate: {
  initialDiff,    // 既存。null許容。明示操作時のみ数値化
  certainty,      // 既存。実戦モード側からは読み書きしない
  loanRate,       // 既存。投資追加（千円）の換算に使用
  investedTotal,  // 累計投資枚数。既定0
  credit,         // 現在クレジット枚数。既定0
  history         // 差枚操作履歴。既定[]
}
```

計算式: `有利区間差枚 = (initialDiff ?? 0) + (credit - investedTotal)`

履歴:

```js
history: [
  {
    id,
    at,
    mode,
    label,
    input,
    before: { initialDiff, investedTotal, credit },
    after:  { initialDiff, investedTotal, credit },
    summary
  }
]
```

- 履歴は `intervalEstimate.history` に保存する
- 保存上限は50件。超過分は古い順に切り捨てる
- シート上の表示は直近10件
- 履歴行からの個別取消は実装しない。直前操作は取り消しバー、過去分は「投資合計を修正」等で復旧する
- 既存データは `history: []` で補完する
- initialDiff は保存上 null を維持し、表示・計算時のみ `initialDiff ?? 0` として扱う
- 既存STEP2の有利区間差枚メモ、timeline の `intervalEvent:'invest'/'credit'`、判定ロジックには触れない

## 5. フィードバック仕様

| イベント | 振動 | 表示 |
|---|---|---|
| 記録成功 | 短1回（10〜30ms） | ボタン押下アニメ＋トースト「◯◯を記録」0.8秒 |
| 取り消し | やや長め1回 | トースト「◯◯を取り消し［元に戻す］」 |
| 無効操作 | 短2回 | トースト（理由を1行） |

- トーストは画面上部中央に表示する
- navigator.vibrate を使用。非対応端末では表示のみで成立
- 音は使用しない
- 数値入力には、単位・換算結果・現在値を添える。暗算や記憶を要求するUIにしない

## 6. 取り消し・区切り・ヤメ仕様

### 6.1 実戦モード取り消し

- 取り消しバーは画面下部に常駐する
- タップで直前イベントを削除。確認ダイアログは出さない
- 削除後、トースト内の［元に戻す］で復元可能
- 対象: counter（G加算含む）、tag、picker、差枚トラッカー操作、差枚スナップショットのログ記録
- 対象外: hit型ウィザードで確定した当選記録、区切りをまたぐ削除、確定済み収支

### 6.2 G数・カウンターの取り消し

- G数加算は加算前の値へ戻す
- カウンターは操作前の値へ戻す
- スイカ履歴・オフセットとの整合を含める

### 6.3 差枚トラッカーの取り消し

- 直前の差枚操作は取り消しバーで操作前の intervalEstimate へ戻す
- 履歴行からの個別取消はしない
- 過去の誤入力は「投資合計を修正」「クレジット更新」「初期差枚設定」「差枚を直接補正」で復旧する

### 6.4 ヤメUI一本化と閉じ忘れガード

- 当選前ヤメは区切り終端 `terminalType: "follow_miss"` として扱う
- 当選前ヤメの表示ラベルは「当選前ヤメ」
- STEP4（ヤメ/収支登録）へ進む時、未終端の区切りがあれば当選前ヤメとして区切るガードを出す
- 新規稼働開始時、未終端の区切りがあれば当選前ヤメとして区切るガードを出す
- これにより「ヤメ」系の出口を timeline メモではなく区切り終端へ統一する

### 6.5 ログ画面の区切り折りたたみ

- ログ画面の各区切り（初当たり単位のブロック）はヘッダタップで開閉可能
- ヘッダには既存のサマリ行と区切りバッジ（1-1等）を維持する
- 初期状態: 当日の稼働の最新区切りのみ開、それ以外はすべて閉
- 「編集」ボタンはヘッダ右端に配置し、開閉状態によらず同じ位置で押せる
- 整理ビュー・PNGカード等、他の出口には影響しない

## 7. 既存コードへの影響範囲（ミコトへの制約）

**変更してよい範囲**

- 実戦モード用のビュー・JSモジュール（接頭辞 bm 等で名前空間分離）
- 稼働入力タブの「実戦モードで記録」ボタン
- 機種プリセットの quickPanel フィールド
- さざなみタグ保存時の entryCause / entryCauseLabel 付与
- intervalEstimate の差枚トラッカー用フィールド（investedTotal / credit / history）
- 当選前ヤメの区切り終端化（terminalType: "follow_miss"）

**変更してはいけない範囲**

- 既存のログデータ構造・localStorage スキーマ（§4.5・§4.6・§6.4 の範囲を除く。追加フィールドは事前相談）
- 既存モーダル・当選記録ウィザードの内部挙動
- スイカオフセット・セグメントライフサイクル・エンディング集計・PNG共有カードの各ロジック
- 既存の稼働入力画面のDOM構造（導線追加箇所を除く）

**実装上の注意**

- 実戦モードオーバーレイより手前に出すべき既存モーダル（テンキー等）の z-index はオーバーレイ（9000）より高くする
- キーボードと共存するシートは visualViewport 追従と復帰時再計算を行う
- 機種プリセットに触れる場合は docs/machine-preset-guide.md の参照と3系統検証を必ず実施する

## 9. 非スコープ（IDEAS控え）

- **配置カスタムUI**: クイックボタンの並べ替え・表示切替。親指可動域は個人差が大きく、固定配置の最適解は存在しないため、最楽ゾーン4枠＋機種別最頻タグ1枠を各自が構成するモデルを検討する
- **有利区間差枚の履歴グラフ**: 区間ごとの推移記録・グラフ化。v1.5では現在値・履歴・任意ログ記録まで
- **ボーナス一括入力シート（B案）**: ボーナス選択→獲得枚数→終了時示唆を実戦モードのボトムシート連鎖で完結させる。現状は既存STEP3誘導強化で対応
- 逐次チップ（記録直後の詳細チップ表示）
- 割り込み復帰（離脱→復帰時のG数更新プロンプト）
- 確認省略/即時記録のモード切替設定
- 進行シーケンス（区切りチェーンの事前定義進行）
- 長押し詳細入力・最近使用履歴・Wake Lock・自動推測
- リプレイ等カウンター実測値を使った分母分析の強化
